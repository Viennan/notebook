# Agent Experience Profile

本文件是本仓库的 AI Agent 经验入口，用于沉淀与用户协作中形成的稳定维护经验。它应控制在 3000 字以内；较长的专题经验放在同目录其他文件，并在这里用链接索引，按需加载。

## 使用原则

- 开始非琐碎维护前，先读本文件；遇到代码分析任务时继续读 [`code-analysis-best-practices.md`](./code-analysis-best-practices.md)。
- 经验写入目标是减少用户未来重复纠正，而不是记录本次任务流水账。
- 只保存稳定、可复用、跨会话仍有价值的经验：用户偏好、仓库维护约定、常见陷阱、验证清单、文档组织方式。
- 不保存短期状态：某次任务完成了什么、临时 TODO、一次性报错、当前分支脏状态、今天的命令输出。
- 不把详细经验写进根 `AGENTS.md`。`AGENTS.md` 只说明 profile 机制；经验正文写在 `profile/EXP.md` 或同目录专题文档。
- `AGENTS.md` 中的 profile 引导应保持通用发现机制；不要让某一种 topic 的经验成为唯一入口。
- 新经验如果会让 `EXP.md` 变长，优先合并旧条目或抽成同目录专题文档再链接。
- 本机制借鉴 Hermes memory/background review prompt 的边界：保存稳定事实、偏好、纠正和工作流经验；把一次性进展、临时任务和可执行长流程排除在外。

## 何时主动总结

完成一次协作后，如果出现以下信号，应考虑更新 profile：

- 用户纠正了 agent 的工作方式、文档风格、范围控制或验证习惯。
- 某个维护流程被反复使用，未来 agent 可直接复用。
- 发现会导致误改、漏更新索引、污染 submodule、破坏文档导航的坑。
- 新增了一个 topic 级最佳实践，例如权威参考资料索引、报告结构、初始化流程。
- 用户明确要求“记住/沉淀/以后按这个来”。

若没有稳定经验，只在最终回复中说明结果，不写 profile。

## 写入格式

优先写成短规则，而不是叙事：

```text
### 主题

- 触发场景：什么时候适用。
- 操作规则：以后怎么做。
- 反例/不要做：避免什么。
- 参考：必要时链接同目录专题文档。
```

## 当前经验索引

### 代码分析 topic

详见 [`code-analysis-best-practices.md`](./code-analysis-best-practices.md)。

核心记忆：

- 新建代码分析 topic 时，使用 `notes/code-ana_${RepoName}/`，其中 submodule 放在 topic 目录内。
- 引入 submodule 的命令交给用户执行；用户确认后，agent 再继续创建索引、记录 commit 和做后续检查。
- 每个代码分析 topic 应维护 `ref/INDEX.md`，集中索引官方或权威资料。
- 分析结论以当前 submodule commit 的源码为准，在线资料用于补充意图和用户可见行为。
- 深入分析前先找主执行链路，再围绕用户问题做专题，不按目录复述源码。
- 代码分析报告先回答 why/what/how：功能目标、领域模型、运行路径；实现代码只作为证据，必要时用流程图或时序图解释。
- 结束前检查 topic `INDEX.md` 是否更新，确认 submodule 没有非忽略改动。

## 维护边界

- 对用户未要求的历史 topic，不做顺手重构。
- 移动经验文档时保留可导航入口，避免旧路径说明继续误导后续 agent。
- 对 profile 经验的更新要像 memory 写入一样克制：高信号、短、可审计。
